# InsFlow: система заявок и сводов

Корпоративный Django-сервис для он-лайн брокера, который автоматизирует работу со страховыми заявками на лизинговое имущество:

- загрузка входящих Excel-заявок;
- извлечение и нормализация данных;
- генерация текста письма по шаблону;
- сбор предложений страховых компаний в свод;
- выгрузка свода в Excel (полная и клиентская версии).

## Что делает проект

Система закрывает полный операционный цикл:

1. Менеджер загружает заявку клиента (`.xls/.xlsx/.xltx`).
2. Сервис парсит Excel и создает страховую заявку с параметрами (тип страхования, франшиза, рассрочка, автозапуск, ДФА, филиал и т.д.).
3. По заявке генерируется текст письма для запроса в страховые компании.
4. После получения ответов формируется свод: предложения добавляются вручную или загружаются из Excel-файлов компаний.
5. Свод переводится по статусам и экспортируется в Excel.

## Ключевые особенности

- Поддержка двух форматов входящих заявок: `КАСКО/спецтехника` и `имущество`.
- Поддержка двух типов шаблона входящей заявки: `юр.лицо` и `ИП` (для ИП применяется смещение строк в парсере).
- Генерация письма без отправки email из системы (кнопка отправки меняет статус заявки).
- Своды с многолетними предложениями, франшизами и рассрочками.
- Загрузка ответов компаний из Excel с валидацией структуры файла.
- Массовая загрузка ответов компаний (несколько файлов за один запрос).
- Экспорт:
  - полный свод (включая технический лист);
  - клиентская версия свода (упрощенная).
- Ролевая модель доступа на группах Django: `Администраторы` и `Пользователи`.
- Поддержка доменной маршрутизации: лендинг на основном домене, рабочее приложение на поддомене.

## Основной workflow

### 1) Заявки

- Страница списка заявок: `/requests/`
- Загрузка новой заявки: `/requests/upload/`
- Поддерживаемые файлы: `.xls`, `.xlsx`, `.xltx` (до 10MB)

Статусы заявки:

- `uploaded` — загружено;
- `email_generated` — письмо сгенерировано;
- `emails_sent` — письма отправлены (статус выставляется вручную).

### 2) Письма

- Генерация темы и тела письма выполняется шаблонизатором.
- Фактическая SMTP-отправка в проекте не используется.

### 3) Своды

- Список сводов: `/summaries/`
- Создание свода из заявки: `/summaries/create/<request_id>/`
- Добавление предложения вручную и редактирование предложений.
- Загрузка ответа одной компании: `.xlsx`.
- Массовая загрузка ответов: до 10 файлов `.xlsx`, до 1MB на файл, суммарно до 10MB.

Статусы свода:

- `collecting` — сбор предложений;
- `ready` — готов к отправке;
- `sent` — отправлен в Альянс;
- `completed_accepted` — завершен: акцепт/распоряжение;
- `completed_rejected` — завершен: не будет.

Автозакрытие просроченных сводов (30 дней от `summary.created_at`):

- Команда: `python manage.py auto_close_stale_summaries`
- Dry-run: `python manage.py auto_close_stale_summaries --dry-run`
- Порог в днях: `python manage.py auto_close_stale_summaries --days=30`

Пример cron (ежедневно в 00:10 МСК):

```cron
10 0 * * * /path/to/project/scripts/cron-auto-close-summaries.sh
```

Если проект запущен в Docker Compose, используйте:

```cron
10 0 * * * USE_DOCKER=1 /path/to/project/scripts/cron-auto-close-summaries.sh
```

## Структура проекта

```text
onlineservice/
├── onlineservice/               # Конфигурация Django (settings, urls, middleware)

├── templates/                   # Django-шаблоны + Excel-шаблоны выгрузки
```

## Технологический стек

- Python 3.11 (Docker), Django 4.2
- PostgreSQL (prod), SQLite (локально по умолчанию)
- pandas + openpyxl + xlrd для Excel
- Bootstrap 5 для интерфейса
- Gunicorn + Nginx + Certbot для production

## Локальный запуск

### 1. Подготовка окружения

```bash
python3 -m venv venv
source venv/bin/activate
pip install -U pip
pip install -r requirements.txt
```

### 2. Настройка переменных окружения

```bash
cp .env.example .env
```

Минимально проверьте:

- `SECRET_KEY`
- `DEBUG`
- `ALLOWED_HOSTS`
- параметры БД (`DB_ENGINE`, `DB_NAME`, ...)

### 3. Инициализация БД и групп

```bash
python manage.py migrate
python manage.py setup_user_groups
```

При необходимости можно создать дефолтных пользователей:

```bash
python manage.py create_default_users --create-test-user
```

### 4. Запуск

```bash
python manage.py runserver
```

Основные URL:

- `http://127.0.0.1:8000/login/`
- `http://127.0.0.1:8000/requests/`
- `http://127.0.0.1:8000/summaries/`

## Важные текущие ограничения

- REST API в проекте сейчас не используется (интерфейс веб-ориентированный).
- Отправка email не выполняется автоматически: письмо генерируется как контент.
- Для загрузки ответов страховых компаний поддерживается только `.xlsx`.

## Лицензия и данные

Проект предназначен для внутреннего использования.

При публикации/передаче репозитория обязательно удаляйте реальные клиентские данные, тестовые логины и артефакты окружения.
