# Выгрузка «Заявка для страховой» (PDF)

## Назначение

В карточке заявки есть две выгрузки, и их важно не путать:

| Выгрузка | Что отдаёт | Формат | Доступ | Кнопка |
|---|---|---|---|---|
| **Карточка заявки** (старая) | ВСЕ поля заявки из БД, включая системные, парсерные, технические (`additional_data`, `source_batch_id`, история, метаданные парсера) | XLSX | Суперпользователь | «Скачать карточку заявки» |
| **Заявка для страховой** (эта фича) | Только подмножество полей, нужное андеррайтеру для расчёта тарифа | PDF | Суперпользователь | «Скачать заявку (PDF)» |

Карточка — это «дамп для отладки». Заявка для страховой — чистый документ, который можно отправить в страховую компанию: ничего лишнего и внутреннего туда не попадает.

Ключевой принцип отбора: **поле включается, если оно влияет на расчёт тарифа / нужно андеррайтеру**. Это этап расчёта, поэтому в документ кладётся «полный пакет» (включая реквизиты страхователя — адреса, вид деятельности), но без внутренней кухни брокера.

## Что попадает в документ

Поля сгруппированы в секции (порядок = порядок в PDF). Строка показывается, только если у неё непустое значение; секция без строк не выводится — поэтому набор полей **контекстный**: КАСКО-заявка покажет блок доп. рисков, имущество — блок имущества, ИП — расширенного страхователя.

- **Заявка**: номер ДФА, филиал, статус сделки, дата подачи, менеджер, срок ответа (МСК).
- **Страхователь**: наименование, ИНН, дата рождения (ИП), юр. адрес, почтовый адрес, основной вид деятельности.
- **Объект страхования**: объект (марка+модель / описание), год выпуска, состояние, тип/категория техники, мощность, стоимость приобретения, количество одинаковых объектов (если > 1), описание объекта.
- **Условия страхования**: тип страхования, срок, территория, франшиза, частота уплаты премии, рассрочка.
- **Дополнительные риски и параметры** (в основном КАСКО/спецтехника): автозапуск, КАСКО кат. C/E, комплектность ключей, ПТС/ПСМ, телематический комплекс, банк-кредитор, цели использования, перевозка (маршрут + срок), СМР.
- **Условия страхования имущества**: страхователь, тип страховой суммы, условия охраны/хранения, правообладатель места расположения.

### Особые правила

- При `franchise_type = both_variants` в строке «Франшиза» добавляется явная просьба: *«просьба рассчитать оба варианта — с франшизой и без»*.
- Перевозка показывается одной строкой `Пункт отправления → Пункт назначения, срок N дн.`, только если `has_transportation = True`.
- Булевы флаги (`has_autostart`, `has_casco_ce`, `has_construction_work`, `has_installment`) показываются как «Да», только когда взведены; `False` = просто отсутствие строки.

### Что НЕ попадает (исключено намеренно)

`id`, `status`, `created_at/by`, `updated_at`, `source_batch_id`, `item_no`, `item_count`, `parser_confidence`, `manual_edits_count`, `additional_data`, история правок, `notes` (внутренние примечания брокера), `email_subject`, сырой `vehicle_info`.

## Как устроена генерация PDF

1. View достаёт `InsuranceRequest` и вызывает `render_application_pdf(insurance_request)`.
2. `build_application_context` прогоняет заявку через манифест `SECTIONS`, оставляя только непустые строки и секции, и добавляет `title`, `deadline`, `generated_at`.
3. Контекст рендерится в HTML через Django-шаблон `application_pdf.html`.
4. `xhtml2pdf.pisa.CreatePDF` превращает HTML в PDF. Кириллица обеспечивается `@font-face` в CSS шаблона + `_link_callback`, который резолвит `url(fonts/DejaVuSans.ttf)` в абсолютный путь внутри `core/fonts/`.
5. `import xhtml2pdf` сделан **внутри** `render_application_pdf`, а не на уровне модуля — отсутствие пакета не ломает импорт `views.py`.

Логика отбора полей (`SECTIONS` + `build_application_context`) отделена от рендера, поэтому при необходимости тот же набор данных можно сериализовать в другой формат (JSON по схеме v2, Excel) без переписывания.

## Технический выбор и подводные камни

**Движок — xhtml2pdf** (а не WeasyPrint / ReportLab): вёрстка обычным HTML-шаблоном, ставится через pip без правки Dockerfile. WeasyPrint потребовал бы системных библиотек (libpango/libcairo) в `python:3.11-slim`; ReportLab — вёрстку кодом.

**Шрифты в репозитории.** reportlab (движок под xhtml2pdf) не имеет кириллицы в стандартных Type1-шрифтах, поэтому DejaVu Sans (свободная лицензия) лежит прямо в `core/fonts/` и подключается через `@font-face`. Альтернатива — ставить `fonts-dejavu-core` через apt в Dockerfile и резолвить системный путь; пока выбран вариант с бинарниками в git (~1.4 МБ).

**`svglib` закреплён на 1.5.1 — это критично для прода.** xhtml2pdf зависит от `svglib`; начиная с 1.6.0 svglib тянет `rlpycairo → pycairo → системный cairo`, которого нет в `python:3.11-slim` (сборка падает на pkg-config). SVG в наших заявках не используется, поэтому держим `svglib==1.5.1` и не тащим cairo. Причина продублирована комментарием в `requirements.txt` — не снимать пин, не разобравшись.
