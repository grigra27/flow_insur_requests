# План улучшения загрузчика заявок: Parser V2

## Цель

Создать новый экспериментальный загрузчик заявок из Excel, который повысит качество распознавания реальных заявок без риска для текущего рабочего процесса сотрудников.

Текущий загрузчик должен продолжить работать в прежнем виде. Parser V2 создается как отдельный сценарий загрузки: отдельная ссылка, отдельная страница, отдельная логика разбора и отдельный пользовательский путь.

Главный принцип Parser V2: заявка должна создаваться всегда, если файл был загружен. Парсер должен забрать из файла все, что смог уверенно распознать, а все сомнительные или нераспознанные данные показать сотруднику для ручной проверки и исправления.

## Жесткие ограничения

1. Старый загрузчик не меняем.

   Действующий сценарий `insurance_requests:upload_excel` должен остаться доступным и работать так же, как сейчас. Его нельзя ломать, переписывать поверх или подменять новой логикой, потому что им уже пользуются сотрудники.

2. Parser V2 делаем с нуля.

   Новый парсер должен быть отдельным сервисом и отдельной страницей. Старый код можно использовать как источник знаний, но не как место для рискованных правок.

3. Доступ только для superuser.

   Ссылку на Parser V2 можно разместить в административной зоне или рядом с блоком аналитики, но сам раздел не должен быть доступен обычным сотрудникам. Базовое правило доступа: `request.user.is_superuser`.

4. Заявка создается даже при неполном распознавании.

   Parser V2 не должен блокировать создание заявки из-за пустого клиента, непонятного объекта, неизвестной франшизы или другого поля. Корректный сценарий: создать заявку с максимально заполненными данными, сохранить предупреждения и дать сотруднику возможность поправить карточку через интерфейс.

5. Ошибки чтения превращаем в понятный рабочий сценарий.

   Если файл частично читается, создаем заявку из того, что удалось извлечь. Если файл совсем не читается, все равно нужно стремиться создать минимальную заявку с привязанным исходным файлом, именем файла, пометкой о проблеме чтения и задачей на ручную обработку. Единственный нерабочий случай: файл фактически не был загружен.

## Предлагаемый пользовательский сценарий

1. Superuser открывает отдельную страницу Parser V2.

   Возможный маршрут: `/insurance-requests/upload-v2/`.

   Возможное имя route: `insurance_requests:upload_excel_v2`.

2. Пользователь загружает Excel-файл заявки.

   На этом шаге файл еще не создает финальную заявку сразу. Сначала он проходит разбор и попадает на страницу предварительной проверки.

3. Parser V2 разбирает файл.

   На выходе формируется нормализованный результат:

   - распознанные поля;
   - список объектов страхования;
   - источник каждого значения в Excel, если его удалось определить;
   - предупреждения по сомнительным, конфликтующим или пустым данным;
   - технические данные разбора для последующего анализа качества.

4. Открывается промежуточная страница проверки.

   На этой странице сотрудник видит, что именно было распознано, что не распознано и какие поля требуют внимания. Все ключевые поля должны быть редактируемыми до создания заявки.

5. Пользователь исправляет данные при необходимости.

   Например, может вручную указать клиента, уточнить объект, поправить срок страхования, франшизу, филиал, менеджера, вид страхования, условия перевозки или список объектов.

6. Пользователь нажимает `Создать заявку`.

   Заявка создается всегда. Даже если часть полей осталась пустой или сомнительной, Parser V2 должен сохранить заявку в текущую модель с максимально возможным заполнением.

7. После создания пользователь попадает в карточку заявки или на страницу редактирования.

   Оптимальный вариант: после создания открыть существующую страницу редактирования заявки, чтобы сотрудник мог сразу доправить данные в привычном интерфейсе.

## Что показывать на странице предварительной проверки

Страница предварительной проверки должна быть не отчетом, а рабочей формой перед созданием заявки.

Рекомендуемые блоки:

1. Основные данные заявки.

   Номер заявки, филиал, менеджер, дата, срок страхования, вид страхования, статус или тип заявки.

2. Клиент и участники.

   Страхователь, выгодоприобретатель, лизингодатель, лизингополучатель, контактные данные, если они есть в файле.

3. Условия страхования.

   Страховая сумма, премия, франшиза, территория, порядок оплаты, периодичность платежей, дополнительные условия.

4. Объекты страхования.

   Таблица объектов с возможностью редактирования строк. Для многообъектных заявок важно поддержать добавление, удаление и правку объектов до создания заявки.

5. Дополнительные условия и особенности.

   Противоугонные системы, телематика, перевозка, хранение, монтаж, имущество, CMR, особые условия по эксплуатации.

6. Предупреждения парсера.

   Список того, что не удалось распознать или что распознано с низкой уверенностью. Предупреждения не должны блокировать кнопку создания заявки.

7. Исходный файл.

   Название файла, дата загрузки, ссылка на сохраненный оригинал после создания заявки.

## Архитектура Parser V2

Parser V2 лучше вынести в отдельный модуль, чтобы его можно было развивать независимо от текущего загрузчика.

Возможная структура:

```text
insurance_requests/
  parsers/
    excel_v2/
      __init__.py
      workbook_reader.py
      normalizer.py
      section_finder.py
      extractors.py
      result.py
      warnings.py
```

Основные компоненты:

1. `WorkbookReader`.

   Открывает `.xls` и `.xlsx`, извлекает листы, ячейки, объединенные области, стили и текстовые значения. На этом уровне не должно быть бизнес-логики заявки.

2. `Normalizer`.

   Приводит текст ячеек к удобному для поиска виду: убирает лишние пробелы, приводит регистр, нормализует переносы строк, двоеточия, кавычки и типовые сокращения.

3. `SectionFinder`.

   Находит смысловые блоки заявки не по жестким координатам, а по подписям и окружению: клиент, объект, условия страхования, франшиза, график платежей, имущество, перевозка.

4. `FieldExtractors`.

   Отдельные извлекатели для групп полей. Например: клиент, филиал, менеджер, сроки, объекты, франшиза, платежи, имущество, транспортировка, противоугонные системы.

5. `ParseResult`.

   Единый результат разбора:

   ```text
   data
   warnings
   source_map
   confidence
   raw_debug
   ```

6. `PreviewForm`.

   Форма, которая строится на основе результата разбора и позволяет сотруднику вручную скорректировать данные перед созданием заявки.

## Принцип распознавания

Текущий риск Excel-загрузчика в том, что реальные заявки похожи друг на друга, но не являются строго одинаковыми. В реальных файлах меняются строки, появляются дополнительные блоки, многообъектные таблицы, имущество, перевозка, нестандартные сроки, дополнительные условия и примечания.

Parser V2 должен перейти от координатного подхода к гибридному:

1. Искать поля по подписям.

   Например, не “ячейка B14”, а “значение рядом с подписью `Страхователь`”.

2. Учитывать окружение.

   Если рядом есть блок `Выгодоприобретатель`, `Лизингополучатель`, `Объект страхования`, парсер должен понимать, к какому смысловому разделу относится значение.

3. Поддерживать табличные блоки переменной длины.

   Особенно важно для многообъектных заявок, где объектов может быть 2, 15 или больше.

4. Не считать пустое поле ошибкой импорта.

   Пустое поле означает предупреждение и последующую ручную правку, а не отказ от создания заявки.

5. Сохранять исходное значение.

   Если значение удалось нормализовать, нужно хранить и нормализованное значение, и исходный текст из Excel. Это поможет разбирать спорные случаи.

## Что улучшить в первую очередь

### 1. Динамическое распознавание объектов

Самое важное улучшение для качества загрузки - корректно обрабатывать объект или список объектов без привязки к фиксированным строкам.

Нужно поддержать:

- один объект;
- несколько объектов;
- длинные многообъектные заявки;
- объекты с частично заполненными строками;
- имущество без стандартных транспортных полей;
- транспортировку имущества как отдельный блок условий.

В текущую модель можно сохранять краткое описание в существующее поле объекта, а полный список объектов класть в технический JSON-блок, например в `additional_data.parser_v2.insured_objects`.

### 2. Страница предварительной проверки

Это даст быстрый практический эффект даже до идеального парсинга. Если Parser V2 распознает 70-80% данных, сотрудник сможет поправить оставшееся до создания заявки, вместо того чтобы потом искать ошибки в уже созданной карточке.

На этой странице важно показывать не только финальные значения, но и предупреждения: “клиент не найден”, “франшиза распознана частично”, “найдено 15 объектов”, “срок страхования не удалось привести к стандартному формату”.

### 3. Сохранение расширенного результата разбора

Даже если текущая модель заявки не поддерживает все поля Excel, Parser V2 должен сохранять расширенный результат в техническое поле.

Это позволит:

- не терять данные из заявки;
- сравнивать качество разных версий парсера;
- улучшать парсер на реальных ошибках;
- в будущем безболезненно перейти на более богатую модель данных.

### 4. Карта источников значений

Для каждого распознанного поля желательно сохранять, откуда оно взято:

```text
branch.name -> Лист1!C4
manager.full_name -> Лист1!C5
insured_objects[0].name -> Лист1!B28
franchise -> Лист1!D18
```

Это сильно упростит отладку: можно быстро понять, почему Parser V2 взял конкретное значение.

### 5. Предупреждения вместо отказов

Parser V2 должен возвращать предупреждения разных уровней:

- `info` - просто зафиксирован факт, например “найдено 15 объектов”;
- `warning` - поле не найдено или распознано с низкой уверенностью;
- `error` - серьезная проблема чтения, но заявка все равно может быть создана как минимальная;
- `manual_required` - значение обязательно требует проверки сотрудником.

При этом ни один из этих уровней сам по себе не должен запрещать создание заявки.

## Сохранение данных в текущую систему

На первом этапе лучше не менять радикально модель заявки. Быстрый и безопасный подход:

1. Существующие поля заявки заполнять максимально совместимо с текущей карточкой.

2. Если поле не распознано, использовать безопасное значение-заглушку или оставлять пустым, если текущая модель это позволяет.

3. Полное содержимое Parser V2 сохранять в `additional_data`.

   Пример структуры:

   ```json
   {
     "parser_v2": {
       "version": "2.0.0",
       "source_file_name": "example.xls",
       "confidence": 0.82,
       "warnings": [],
       "source_map": {},
       "parsed_payload": {}
     }
   }
   ```

4. Оригинальный Excel-файл обязательно сохранять вместе с заявкой.

5. Если заявка создана с проблемами распознавания, добавить в карточку заметку или внутренний признак, чтобы сотрудник видел необходимость проверки.

## Этапы реализации

### Этап 1. Каркас V2 без вмешательства в старый загрузчик

Результат этапа:

- отдельный URL для Parser V2;
- доступ только для superuser;
- отдельная форма загрузки файла;
- базовый сервис разбора;
- простая preview-страница;
- старый загрузчик не изменен.

Это минимальный безопасный фундамент.

### Этап 2. Preview-страница и создание заявки

Результат этапа:

- Parser V2 показывает распознанные поля;
- сотрудник может исправить данные перед созданием;
- кнопка создания заявки доступна всегда;
- заявка создается из исправленных данных;
- предупреждения и технический результат сохраняются в `additional_data`;
- после создания пользователь попадает в редактирование или карточку заявки.

### Этап 3. Улучшение распознавания по реальным заявкам

Результат этапа:

- динамическое распознавание объектов;
- поддержка многообъектных заявок;
- поддержка имущества и перевозки;
- лучшее распознавание франшиз, сроков, платежей и дополнительных условий;
- source map для отладки.

На этом этапе нужно использовать уже собранную подборку реальных заявок из `avtozayavka/real_requests` как тестовый корпус.

### Этап 4. Метрики качества

Результат этапа:

- фиксируем количество предупреждений по каждой загрузке;
- считаем, какие поля чаще всего не распознаются;
- сравниваем результат Parser V2 с ручными исправлениями сотрудника на preview-странице;
- постепенно повышаем точность парсера на основании реальной статистики.

### Этап 5. Расширение доступа

Только после проверки качества Parser V2 можно обсуждать доступ не только для superuser, но и для ограниченной группы сотрудников. До этого V2 лучше считать экспериментальным инструментом.

## Тестирование

Для Parser V2 нужен отдельный набор тестов на реальных заявках.

Минимальный набор:

1. Простая заявка с одним объектом.

2. Крупная многообъектная заявка.

3. Заявка по имуществу.

4. Заявка с перевозкой.

5. Заявка с нестандартным сроком страхования.

6. Заявка с частично пустыми полями.

7. Файл, который читается с ошибками или нестандартной структурой.

Для каждого файла желательно хранить ожидаемый результат в JSON-фикстуре. Тесты не должны требовать стопроцентного совпадения всех второстепенных полей на первом этапе, но должны проверять ключевые поля и отсутствие падения парсера.

## Критерии готовности первой версии

Parser V2 можно считать готовым к внутреннему тестированию, если:

1. Старый загрузчик работает без изменений.

2. Новый загрузчик доступен по отдельной ссылке только superuser.

3. Любой валидный загруженный Excel приводит либо к созданию заявки, либо к созданию минимальной заявки с понятной пометкой о проблеме чтения.

4. Пользователь видит preview-страницу перед созданием заявки.

5. На preview-странице можно исправить ключевые поля.

6. Многообъектная заявка не ломает импорт.

7. Оригинальный файл и технический результат разбора сохраняются.

8. Предупреждения парсера видны пользователю и сохраняются для анализа.

## Рекомендуемый первый инкремент

Самый полезный быстрый инкремент:

1. Добавить отдельную страницу Parser V2 только для superuser.

2. Сделать загрузку Excel и базовый разбор без создания заявки.

3. Показать preview-страницу с распознанными полями и предупреждениями.

4. Добавить создание заявки из preview-формы.

5. Сохранить весь результат разбора в `additional_data`.

После этого можно улучшать сами extractors, не трогая пользовательский сценарий и не рискуя старым загрузчиком.

## Вопросы для уточнения перед разработкой

1. Где лучше разместить ссылку на Parser V2: в списке заявок, в административном меню или рядом с аналитикой?

2. Нужно ли после создания заявки открывать карточку или сразу страницу редактирования?

3. Какие поля считаются обязательными для ручной проверки на preview-странице?

4. Нужно ли помечать созданные через Parser V2 заявки отдельным признаком в интерфейсе?

5. Нужно ли в первой версии показывать сотруднику source map, или оставить ее только в технических данных?
