# Parser V2: распознавание заявок по имуществу

Дата создания: 2026-06-04.
Статус: **черновик плана**.

## Контекст

Parser V2 готовится к внедрению на замену V1 (`core/excel_utils.py`).
На текущей выборке заявок КАСКО/спецтехника V2 работает хорошо, но на
заявках по имуществу (`avtozayavka/im_request/*.xls`, 10 файлов из реальной
почты) разбор почти всегда даёт ошибочный результат. Главная ошибка
видна сразу: вместо одного реального объекта страхования парсер
возвращает **6 объектов** (1 настоящий + 5 фейков из блока
«Дополнительные виды страхования») и склеивает их в `vehicle_info`.

Поскольку при создании заявок V2 разделяет файл на N сестринских
`InsuranceRequest` рядов (по одному на объект, см.
`one_request_per_object`), эта ошибка означает: из одной имущественной
заявки V2 создаст **6 запросов** и отправит **6 писем** страховщикам.
Это блокер на внедрении V2.

## Структура имущественной формы (реверс-инжиниринг 10 файлов)

Layout стабильный по всем 10 файлам (`avtozayavka/im_request/`). Между
КАСКО и имуществом форма различается существенно:

```
row 2  : B='Заявка на страхование №', H='ОБ-<номер>-<филиал>', L='дата подачи', M=<дата>
row 4  : B='Подразделение', C='<филиал>'                     ← филиал может НЕ содержать слово «филиал» (напр. «Санкт-Петербург»)
row 7  : B='Наименование Лизингополучателя:', D='<ООО ...>'   ← для ИП-формы D7='ИП <ФИО>' либо просто 'ИП '
row 8  : (юр.лицо) B='Юридический адрес:'  /  (ИП) B='дата рождения', D='<dd.mm.yy>'
row 22 : B='Страхование имущества предприятий'               ← label-only, БЕЗ X-mark
row 39 : B='СВЕДЕНИЯ ОБ ОБЪЕКТЕ СТРАХОВАНИЯ'                  ← заголовок раздела (для ИП — B40)
row 41 : шапка таблицы — B='№ п/п', C='Наименование...', K='Год выпуска', M='Стоимость', N='Валюта'  (для ИП — row 42)
row 42+: сами объекты — B=№, C=описание, K=год, L=новое/б.у, M=цена, N=валюта, P=экв. в другой валюте
row 44+: «Дополнительные виды страхования оборудования (отметьте Х)»
         + C44='Х', D44='Перевозка (с погрузкой, выгрузкой) от поставщика к лизингополучателю'
         + C45/D45 'Пункт отправления'   <значение>
         + C46/D46 'Пункт назначения'    <значение>
         + C47/D47 'Ориентировочный срок перевозки, в днях'   <значение>
         + D48     'Страхование монтажных (строительно-монтажных) работ ...'
```

Отличия от КАСКО, существенные для парсера:

- **Описание объекта** — только колонка **C** (а не C..I, как в КАСКО).
- **Колонки финансов сдвинуты на +1** относительно КАСКО:
  год **K** (а не J), состояние **L** (а не K), цена **M** (или L в
  некоторых casco-шаблонах), валюта **N**.
- **Тип страхования** определяется одним label-ом в **B22** (без
  X-метки), а не X-меткой в `D21/D22/D23`.
- **DFA-номер** в `H2` всегда имеет буквенный префикс «ОБ-» (у КАСКО
  и спецтехники префикса нет).
- **Филиал** в `C4` может быть городом без слова «филиал» (напр.
  «Санкт-Петербург» в файле 20703).

## Корневые причины

1. **Нет stop-маркера для конца таблицы объектов** (`parser.py:1968-1980`,
   `_is_object_stop_row`). В списке остановки нет фраз
   «дополнительные виды страхования», «перевозка», «пункт отправления»,
   «пункт назначения», «срок перевозки», «строительно-монтаж». Цикл
   в `_extract_objects` (`parser.py:1084`) `for row_number in range(start_row, start_row + 35)`
   уходит вниз на 35 строк и проглатывает блок дополнительного
   страхования как объекты. **Это источник 5 фейков на файл.**

2. **`description` объекта = `row_text` целиком** (`parser.py:1242`,
   `_build_object_payload`). Туда попадают год, цена, валюта вместе с
   наименованием. На выходе `_vehicle_summary` склеивает description'ы
   всех объектов в одну строку через `\n` — получается мусор вроде
   `'1 Комплект спортивного оборудования 2024 новое 32485200 руб экв 314 079 USD\nДополнительные виды страхования...'`.
   Это поле уйдёт в email страховщикам.

3. **`asset_status` для имущества подхватывает год выпуска**
   (`parser.py:1401-1442`, `_extract_asset_status`). Fallback по
   координатам: строки 43/45/47/49, колонка **K (11)**. Для КАСКО
   `K` — состояние; **для имущества `K` — это год выпуска**, состояние
   лежит в `L`. Эффект: файл 20506 даёт `asset_status='2025 2025 2025'`
   (читает три K-ячейки с годами).

4. **`_looks_like_client_name` пропускает дату как имя для ИП-имущества**
   (`parser.py:1927-1940`). В файле 19770 `D7='ИП '` (длина 2 →
   отсеяно), `D8='12.06.80'` — нормализуется в `'12 06 80'`, длина 8,
   не в blocked_values, нет «адрес» → True. Парсер берёт дату как имя
   клиента. Нужна проверка на наличие букв / отсутствие чисто-датового
   формата.

5. **dfa_number regex теряет префикс «ОБ-…» и обрывается на underscore**
   (`parser.py:916`, `_extract_dfa_number`). Регексп
   `\b\d{4,6}(?:[-/][0-9A-Za-zА-Яа-яЁё]+){1,5}\b` требует:
   - чисто-цифровой первой группы 4–6 знаков (отбрасывает префикс «ОБ»
     в `ОБ-20703-ЛО`),
   - разделитель только `[-/]` (теряет хвост после `_` в именах файлов
     `Заявка_ОБ_20703-ЛО_Невский...`),
   - минимум один разделитель в номере (отбрасывает чистое `20009`).
   Эффекты: 20009 → не распознан, 20703 → `20703-ЛО` (обрезано), 20433
   → `20433-ЛО` (обрезано).

6. **branch не распознан для нестандартного формата филиала**
   (`parser.py:902-913`, `_extract_branch`). Ищет вхождение «филиал»
   или «обособленное подразделение». Файл 20703: `C4='Санкт-Петербург'`
   — слово «филиал» отсутствует → branch пустой.

7. **manufacturing_year fallback берёт дату подачи или дату договора**
   (`parser.py:1385-1394`). Fallback `_year_from_text` пробегает по
   всем ячейкам и возвращает первое попавшееся `19XX|20XX`. Часто это
   оказывается **год из M2 (дата подачи)** или **M15 (дата начала
   договора)**, а не год выпуска объекта.

## Что НЕ ломается (вопреки ожиданиям)

- `insurance_type='страхование имущества'` корректно через
  label-fallback в `_extract_insurance_type_from_block_rows`: находит
  `B22='Страхование имущества предприятий'` и матчит подстроку
  «имуществ».
- `application_format='property'` корректно во всех 10 файлах.
- Первый, реальный объект (brand/model/year/condition/cost/currency)
  распознаётся корректно во **всех 10 файлах**.
- Identical-object multiplicity работает (20399: 2 → count=2, 20506:
  6 → count=6).
- `has_transportation` корректно `True` там, где `C44='Х'` рядом с
  «Перевозка» (20009, 20061, 20165).
- ИНН — везде корректно.

## План работ

### Этап 1 — stop-маркеры (минимальный фикс, разблокирует выкатку)

Цель: убрать 5 фейк-объектов на файл, привести `objects` к 1.

- В `_is_object_stop_row` (`parser.py:1968`) добавить:
  ```
  "дополнительные виды страхования",
  "перевозка с погрузкой",
  "пункт отправления",
  "пункт назначения",
  "ориентировочный срок перевозки",
  "строительно-монтажн",
  "монтажн",
  ```
- Acceptance: на всех 10 файлах `avtozayavka/im_request/`
  `len(insured_objects) == 1` (или count multiplicity, если в файле
  несколько одинаковых строк, как в 20506/20399/20433). Никаких
  «Перевозка», «Пункт отправления», «СМР» в `insured_objects`.
- Тест: добавить в `insurance_requests/test_parser_v2.py` сценарий
  «property-form does not slurp дополнительные виды страхования» с
  fixture-файлом из `avtozayavka/im_request/Заявка__20543_-_Обсадной_стол.xls`
  (или сделать минимальный синтетический Excel).

### Этап 2 — спецветка для property в `_extract_objects`

Цель: убрать фундаментальную зависимость от КАСКО-layout, сделать
parsing объектов имущества явным.

После того как `_detect_application_format(data) == 'property'`, идти
по специализированному пути:

- start_row = первая строка, где **B содержит «№ п/п»** и в строке есть
  «Наименование» (это всегда B41 для юр.лица / B42 для ИП).
- объекты — только строки сразу под шапкой, где **B содержит число и
  C непусто**.
- стоп по первому пустому B **или** по строке с любым из stop-маркеров
  этапа 1.
- layout фиксированный: `C=наименование`, `K=год`, `L=состояние`,
  `M=цена`, `N=валюта`.
- `description` собирать **только из C** (не из row_text).

Acceptance: на всех 10 файлах `description` каждого объекта содержит
только наименование без года/цены/валюты; `_vehicle_summary` в email
читаемый.

### Этап 3 — asset_status для property

В `_extract_asset_status` (`parser.py:1401-1442`):

- Если `application_format == 'property'`, fallback по координатам идёт
  по **колонке L (12)**, а не K. Либо вовсе отключить fallback и
  оставить только label-based (для имущества состояние не критично, в
  типовом шаблоне отдельного поля «Состояние» вообще нет — оно
  совпадает с L42).
- Acceptance: файл 20506 даёт `asset_status` пустым или 'новое', не
  `'2025 2025 2025'`.

### Этап 4 — _vehicle_summary без мусора

`_vehicle_summary` (`parser.py:1842-1848`) сейчас склеивает
`description` объектов через `\n`. После этапа 2 `description` уже
чистое, но имеет смысл явно собирать строку из
`f"{brand} {model} {year} {cost} {currency}"` либо использовать
очищенный description.

Acceptance: `vehicle_info` в email — это читаемый список «<наименование>,
<год>, <стоимость> <валюта>» без «Дополнительные виды страхования», без
«Х», без `(отметьте знаком "Х")`.

### Этап 5 — dfa_number regex

В `_extract_dfa_number` (`parser.py:915-924`):

- расширить паттерн под опциональный буквенный префикс:
  `(?:[A-Za-zА-Яа-яЁё]{1,3}[-_]?)?(\d{4,6}(?:[-_/][A-Za-zА-Яа-яЁё0-9]+)*)`;
- в filename-fallback нормализовать `_` в `-` перед матчем — это даст
  правильное распознавание `Заявка_ОБ_20703-ЛО_Невский...` →
  `ОБ-20703-ЛО`;
- разрешить чисто-цифровой номер (`20009` без хвоста) — иначе файл
  20009 теряет DFA полностью.

Acceptance: на всех 10 файлах `dfa_number` непустой и совпадает с
номером в `H2` исходного файла (включая префикс `ОБ-`, если он есть).

### Этап 6 — branch fallback для property

В `_extract_branch` (`parser.py:902-913`):

- для `application_format == 'property'` добавить fallback по
  координате C4 (по аналогии с CLIENT_COORDINATES для D7);
- значение из C4 пропускать через `map_branch_name` — для «Санкт-Петербург»
  существующий маппинг `BRANCH_MAPPING` в `core/excel_utils.py:11`
  уже должен дать корректное короткое имя.

Acceptance: файл 20703 даёт `branch='Санкт-Петербург'` (или то, что
выдаёт `map_branch_name` для этого ввода — проверить).

### Этап 7 — `_looks_like_client_name` отсеивает даты

В `_looks_like_client_name` (`parser.py:1927-1940`):

- отбрасывать значения, состоящие только из цифр и разделителей `.-/`,
  например `re.fullmatch(r'[\d.\-/]+', raw)` → False;
- файл 19770 после этого: `D7='ИП '` короче 3 → False, `D8='12.06.80'`
  → False (дата). Парсер не подставит дату как имя.

Acceptance: файл 19770 `client_name` либо равен `'ИП '` (как есть в
файле, оператор дозаполнит вручную), либо `MISSING_CLIENT`, но **не
`'12.06.80'`**.

### Этап 8 — manufacturing_year только в пределах таблицы объектов

В `_extract_manufacturing_year` (`parser.py:1385-1394`):

- отключить «глобальный» fallback по всем ячейкам;
- если есть таблица объектов, брать `year` первого объекта из
  `insured_objects[0].year`;
- иначе оставить пустым.

Acceptance: ни в одном из 10 файлов `manufacturing_year` не равен году
из M2 (дата подачи) или M15 (дата начала договора).

## Минимальная итерация для unblock

Этапы 1 + 2 + 3 достаточны, чтобы V2 на всех 10 файлах давал ровно
один корректный объект и `application_format='property'` без фейков.
Этапы 4–8 — последовательные улучшения качества полей, не блокеры на
выкатке.
