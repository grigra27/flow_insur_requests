# Собственная форма страховой заявки

Дата: 2026-05-18.
Статус: верхнеуровневый план, до проектирования и кода.

## Цель

Перестать прикладывать к письму страховщикам исходный Excel лизинговой
компании и начать рассылать **наш собственный документ** заявки, который
система генерирует из нормализованных данных. Лизинговая заявка становится
только источником данных, а не самим вложением.

## Текущее положение

- В письмо страховщикам сейчас прикладывается оригинальный Excel от лизинга
  (тот же, что мы парсим).
- В `InsuranceRequest` есть основные поля (клиент, ИНН, тип, период,
  vehicle_info, ДФА, филиал, франшизы, флаги, телематика, ключи, ПТС/ПСМ,
  банк-кредитор, цели, год, статус имущества, менеджер, статус сделки)
  + `additional_data` JSON.
- Парсер `core/excel_utils.py` читает по координатам ячеек; `parsers/excel_v2`
  начат как более устойчивая ветка ([insurance_request_parser_v2_improvement_plan.md](../insurance_request_parser_v2_improvement_plan.md)).
- Уже спроектирован машиночитаемый контракт собственной заявки —
  [insurance_request_format_package/](../insurance_request_format_package/):
  JSON Schema v1 `alliance.insurance_request.v1`, research по 151 файлу,
  три эталонных примера.

## Главный риск (его проверяем первым)

**Не все данные из исходного Excel попадают в БД.** Research уже показывает
расхождение: в схеме v1 есть юридический/почтовый адрес, вид деятельности,
даты договора лизинга, валюта, объект страхования с разобранными
характеристиками — а в модели `InsuranceRequest` этих полей либо нет, либо
они слиты в текстовый `vehicle_info`. До любого PDF нужно убедиться, что
БД способна хранить всё, что есть в источнике.

## Ключевой архитектурный принцип: один объект = одна заявка

В V2-флоу **одна `InsuranceRequest` хранит ровно один объект страхования**.
Если присылаемый Excel содержит N объектов, V2 создаёт N отдельных записей
`InsuranceRequest`. Общие реквизиты (клиент, ИНН, тип/период страхования,
франшиза, ДФА, филиал, менеджер, банк-кредитор, даты лизинга) дублируются
в каждую запись; уникальны только объектные поля (марка, модель, VIN,
год выпуска, состояние, стоимость, валюта и т.д.).

Сёстры одной партии связываются полями:

- `source_batch_id` — UUID, одинаковый у всех заявок, созданных из одного
  Excel.
- `item_no` (1-based) и `item_count` — порядок и общий размер партии.
- В display-имени выводится «ДФА X / объект K из N» при `item_count > 1`;
  одиночные заявки отображаются как раньше.

Каждая заявка живёт **независимо**: свой статус, свой свод котировок, свой
выбранный страховщик, **один PDF и один JSON на заявку**, **одно письмо
страховщику на одну заявку**. Партия — это связка для UI и отчётности, не
сводная единица жизненного цикла.

V1 (`core.excel_utils.ExcelReader`) продолжает работать в режиме
«1 файл → 1 заявка» и не разбивает многообъектные файлы. Splitting
применяется **только к новым загрузкам через V2**. Исторические V1-заявки
не разбиваем постфактум — отображаются как раньше.

## Четыре рабочих потока

### 1. Аудит покрытия Excel → БД (делаем первым)

- Прогнать существующий парсер по всему корпусу `avtozayavka/real_requests`
  + `im_request` и собрать таблицу «поле в исходнике → поле в модели →
  процент заполненности → формат».
- Найти поля, которые есть в Excel, но **никуда не попадают**, и поля,
  которые сейчас слиты в `vehicle_info` / `additional_data` (фактически
  потеряны для структурного использования).
- Сверить результат с JSON Schema v1 — что из схемы у нас в БД не лежит.
- Результат — короткий отчёт `coverage_audit.md` рядом с этим планом:
  список пробелов и рекомендация по расширению модели/JSON-полей.

### 2. Канонический DTO и расширение модели

- Принять JSON Schema v1 как канонический контракт между парсером и
  остальной системой. DTO в коде — pydantic-модель, сгенерированная из
  схемы или написанная вручную с такой же структурой. В схеме блок
  «объект страхования» — **`insured_object` (singular)**, не массив:
  один JSON = одна заявка = один объект.
- По итогам аудита (поток 1) добавить **в `InsuranceRequest`** (не в
  связанную таблицу!) недостающие поля:
  - **Поля партии**: `source_batch_id`, `item_no`, `item_count`.
  - **Поля объекта** (живут прямо в `InsuranceRequest`): `brand`, `model`,
    `vin`, `serial_number`, `manufacturing_year_int`, `condition`
    (`new`/`used`/`unknown`), `equipment_type`, `power_or_capacity`,
    `quantity`, `acquisition_cost_value`, `acquisition_cost_currency`.
  - **Реквизиты страхователя**: `legal_address`, `postal_address`,
    `business_activity`, `ogrn`, `kpp`, `birth_date` (для ИП).
  - **Параметры сделки**: `contract_start_date`, `contract_end_date`,
    `insured_party`, `premium_frequency`, условия покрытия
    (`insured_sum_type`, `indemnity_basis`, `guard_conditions`,
    `property_location_right_holder`).
  - **Расширенные параметры**: structured-франшиза (`franchise_options`
    как JSON-список с фиксированной формой), `anti_theft_*`
    (alarm/immobilizer/satellite — flat-поля или один JSON-объект),
    детали `transportation_origin/destination`, `construction_work_description`.
- Решить, что хранить нормализованно, а что — в `additional_data` JSON
  (правило: всё, что используется в фильтрах/отчётах/PDF — нормализованно;
  свободные примечания и редкие поля — JSON).
- **Отдельной таблицы `InsuredObject` не делаем.** Один объект на одну
  запись — поля объекта живут прямо в `InsuranceRequest`. Это
  принципиальное решение, см. раздел «Ключевой архитектурный принцип».

### 3. Исходящая заявка — наш документ

- **Формат: PDF**, рендеримый из Django HTML-шаблона через WeasyPrint.
  Не редактируется получателем, можно брендировать, легко итерировать
  дизайн. Версия шаблона в footer + в БД.
- Рядом с PDF к письму прикладывается **JSON-приложение** по схеме v1.
  Это будущий «API без API» — крупные страховщики смогут парсить.
- XLSX-вариант делаем только если конкретный страховщик настойчиво
  попросит; тот же DTO рендерится во второй шаблон.
- Раскладка PDF: шапка (логотип, № заявки, дата, срок ответа МСК, менеджер) →
  страхователь → параметры страхования → объект(ы) → особые условия →
  лизинговая сделка → примечания → подвал с реквизитами брокера и
  версией шаблона.
- В письме оставить опцию «приложить оригинал заявки лизинга» —
  для внутреннего архива и для страховщиков, которые попросят.

### 4. Парсер: классификация шаблонов + адаптеры

Долгосрочное направление, можно вести параллельно с потоками 2–3.
Не блокирует первый рассылочный PDF.

- Канонический DTO (поток 2) делает архитектуру парсера осмысленной:
  на выходе любого адаптера — один и тот же DTO.
- Классификатор шаблона по сигнатуре листа (якорные ячейки) → выбор
  специализированного адаптера. Fallback — текущая логика fuzzy-поиска
  по заголовкам.
- LLM-fallback (Claude API) — только для нераспознанных шаблонов,
  с подтверждением оператора. Не на горячем пути.
- Корпус из 174 файлов — регрессионный тест: прогон при каждом
  изменении парсера, диф против эталонного DTO.
- Уже работающий Parser V2 ([insurance_request_parser_v2_improvement_plan.md](../insurance_request_parser_v2_improvement_plan.md))
  — каркас для этого потока.

## Совместимость парсеров V1 и V2

V1 сейчас в проде и каждый день создаёт заявки. V2 — экспериментальный
путь, доступный только superuser через `/upload-v2/`. Расширение модели
не должно ломать V1 и не должно блокировать развитие V2. Стратегия:

### Принцип nullable + двухуровневое заполнение

Каждое новое поле модели добавляется как `null=True, blank=True`.
Парсеры заполняют ровно столько, сколько умеют:

| Категория полей                                | V1                                               | V2                                |
|------------------------------------------------|--------------------------------------------------|-----------------------------------|
| Существующие (`client_name`, `inn`, …)         | заполняет (как сейчас)                           | заполняет                         |
| Новые скалярные (адреса, ОГРН/КПП, даты лизинга, `insured_party`, частота рассрочки, основа возмещения) | оставляет `null` | заполняет, если нашёл             |
| Поля объекта (`brand`, `model`, `vin`, `acquisition_cost`, …) | оставляет `null`                  | заполняет из разобранной строки   |
| Поля партии (`source_batch_id`, `item_no`, `item_count`) | `null`                                  | заполняет; при N=1 — `item_count=1`, `item_no=1` |
| Structured-франшиза, `anti_theft_*`            | `null`                                           | заполняет                         |

V1 продолжает работать **без логических изменений** — «1 файл → 1 заявка»,
многообъектные файлы по-прежнему лепятся в `vehicle_info`. Это его
исторический режим, мы его не трогаем.

### Splitting в V2: 1 файл → N заявок

V2 при обнаружении N объектов в файле:

1. Парсит общие реквизиты (один раз).
2. Парсит массив объектов из файла.
3. Генерирует один `source_batch_id` (UUID) для всей партии.
4. Показывает оператору **превью со списком N будущих заявок** — он может
   отредактировать каждую или отбраковать.
5. На confirm создаёт N записей `InsuranceRequest` с общим `source_batch_id`
   и индексами `item_no=1..N`, `item_count=N`. Общие реквизиты дублируются;
   объектные поля уникальны для каждой записи.

Каждая полученная запись — самостоятельная единица: свой статус, свой
свод, свой PDF/JSON, **своё письмо страховщику**.

### Display-name и UI

- `get_display_name()` возвращает «ДФА X / объект K из N» при
  `item_count > 1`, иначе — текущий формат (ДФА X или #id).
- Список заявок может опционально показывать сёстры партии рядом
  (фильтр / группировка по `source_batch_id`).
- Старые V1-заявки не имеют `item_count`/`item_no` — отображаются как
  раньше.

### Существующие заявки в БД

**Ничего не делаем.** Исторические V1-заявки остаются как одна запись
со слипшимся `vehicle_info` — отображаются так же, как и раньше. Никакой
data-миграции, никакого ретроспективного splitting'а. Splitting применяется
только к новым загрузкам через V2.

### Стратегия перехода

1. **Сейчас** — V1 дефолт, V2 за `/upload-v2/` для superuser.
2. **После расширения модели** — V1 продолжает работать как «1 файл → 1
   заявка», V2 учится splitting'у и заполняет больше полей.
3. **Когда V2-покрытие стабильно ≥ V1 по всем полям и splitting протестирован**
   — сделать V2 дефолтным путём загрузки, V1 оставить как fallback.
4. **После периода стабильности** (несколько месяцев, accept-аудит) — удалить
   V1, освободить `core/excel_utils.py` от роли парсера.

### Что НЕ нужно делать

- Не пытаться «доучить» V1 извлекать новые поля или делать splitting.
  V1 — фиксированные координаты, его расширение хрупкое и съест время,
  которое лучше потратить на V2.
- Не делать новые поля `NOT NULL` — сломает V1-флоу.
- Не вынуждать UI требовать заполнения новых полей. Пустое поле — валидное
  состояние; оператор дозаполняет руками, когда нужно.
- Не разбивать исторические V1-заявки задним числом.
- Не делать отдельной таблицы `InsuredObject` — поля объекта живут
  прямо в `InsuranceRequest`.

## Открытые вопросы

- Юридическая сторона: можно ли брокеру «упаковывать» данные лизинга в
  собственный документ под своим брендом — проверить условия договоров.
- Подпись/печать: ставить ли факсимильную подпись брокера на PDF.
- ~~Многообъектные заявки~~ — решено: один объект = одна заявка
  (см. «Ключевой архитектурный принцип»); многообъектный Excel
  превращается в партию из N заявок, по PDF на каждую.
- Совместимость со страховщиками: кому-то может быть критично сохранить
  и оригинал лизинга. Опция в форме отправки решает.
- Пакетные действия в UI: насколько глубоко поддерживать «выбрать всю
  партию и сделать X» (одобрить, отбраковать, отправить). На старте
  можно ограничиться визуальной группировкой по `source_batch_id`.
