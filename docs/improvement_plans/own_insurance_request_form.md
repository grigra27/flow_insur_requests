# Собственная форма страховой заявки

Дата: 2026-05-18.
Статус: верхнеуровневый план, до проектирования и кода.

## Цель

Перестать прикладывать к письму страховщикам исходный Excel лизинговой
компании и начать рассылать **наш собственный документ** заявки, который
система генерирует из нормализованных данных. Лизинговая заявка становится
только источником данных, а не самим вложением.

## Текущее положение

- В письмо страховщикам сейчас прикладывается оригинальный Excel от лизинга
  (тот же, что мы парсим).
- В `InsuranceRequest` есть основные поля (клиент, ИНН, тип, период,
  vehicle_info, ДФА, филиал, франшизы, флаги, телематика, ключи, ПТС/ПСМ,
  банк-кредитор, цели, год, статус имущества, менеджер, статус сделки)
  + `additional_data` JSON.
- Парсер `core/excel_utils.py` читает по координатам ячеек; `parsers/excel_v2`
  начат как более устойчивая ветка ([insurance_request_parser_v2_improvement_plan.md](../insurance_request_parser_v2_improvement_plan.md)).
- Уже спроектирован машиночитаемый контракт собственной заявки —
  [insurance_request_format_package/](../insurance_request_format_package/):
  JSON Schema v1 `alliance.insurance_request.v1`, research по 151 файлу,
  три эталонных примера.

## Главный риск (его проверяем первым)

**Не все данные из исходного Excel попадают в БД.** Research уже показывает
расхождение: в схеме v1 есть юридический/почтовый адрес, вид деятельности,
даты договора лизинга, валюта, массив объектов с пер-объектными
характеристиками — а в модели `InsuranceRequest` этих полей либо нет, либо
они слиты в текстовый `vehicle_info`. До любого PDF нужно убедиться, что
БД способна хранить всё, что есть в источнике.

## Четыре рабочих потока

### 1. Аудит покрытия Excel → БД (делаем первым)

- Прогнать существующий парсер по всему корпусу `avtozayavka/real_requests`
  + `im_request` и собрать таблицу «поле в исходнике → поле в модели →
  процент заполненности → формат».
- Найти поля, которые есть в Excel, но **никуда не попадают**, и поля,
  которые сейчас слиты в `vehicle_info` / `additional_data` (фактически
  потеряны для структурного использования).
- Сверить результат с JSON Schema v1 — что из схемы у нас в БД не лежит.
- Результат — короткий отчёт `coverage_audit.md` рядом с этим планом:
  список пробелов и рекомендация по расширению модели/JSON-полей.

### 2. Канонический DTO и расширение модели

- Принять JSON Schema v1 как канонический контракт между парсером и
  остальной системой. DTO в коде — pydantic-модель, сгенерированная из
  схемы или написанная вручную с такой же структурой.
- По итогам аудита (поток 1) добавить в `InsuranceRequest` (или в связанные
  таблицы) недостающие поля: страховая сумма и валюта, массив объектов
  страхования с пер-объектной структурой, юр./почтовый адреса, ОГРН/КПП,
  даты договора лизинга, вид деятельности, периодичность рассрочки.
- Решить, что хранить нормализованно, а что — в `additional_data` JSON
  (правило: всё, что используется в фильтрах/отчётах/PDF — нормализованно;
  свободные примечания и редкие поля — JSON).

### 3. Исходящая заявка — наш документ

- **Формат: PDF**, рендеримый из Django HTML-шаблона через WeasyPrint.
  Не редактируется получателем, можно брендировать, легко итерировать
  дизайн. Версия шаблона в footer + в БД.
- Рядом с PDF к письму прикладывается **JSON-приложение** по схеме v1.
  Это будущий «API без API» — крупные страховщики смогут парсить.
- XLSX-вариант делаем только если конкретный страховщик настойчиво
  попросит; тот же DTO рендерится во второй шаблон.
- Раскладка PDF: шапка (логотип, № заявки, дата, срок ответа МСК, менеджер) →
  страхователь → параметры страхования → объект(ы) → особые условия →
  лизинговая сделка → примечания → подвал с реквизитами брокера и
  версией шаблона.
- В письме оставить опцию «приложить оригинал заявки лизинга» —
  для внутреннего архива и для страховщиков, которые попросят.

### 4. Парсер: классификация шаблонов + адаптеры

Долгосрочное направление, можно вести параллельно с потоками 2–3.
Не блокирует первый рассылочный PDF.

- Канонический DTO (поток 2) делает архитектуру парсера осмысленной:
  на выходе любого адаптера — один и тот же DTO.
- Классификатор шаблона по сигнатуре листа (якорные ячейки) → выбор
  специализированного адаптера. Fallback — текущая логика fuzzy-поиска
  по заголовкам.
- LLM-fallback (Claude API) — только для нераспознанных шаблонов,
  с подтверждением оператора. Не на горячем пути.
- Корпус из 174 файлов — регрессионный тест: прогон при каждом
  изменении парсера, диф против эталонного DTO.
- Уже работающий Parser V2 ([insurance_request_parser_v2_improvement_plan.md](../insurance_request_parser_v2_improvement_plan.md))
  — каркас для этого потока.

## Совместимость парсеров V1 и V2

V1 сейчас в проде и каждый день создаёт заявки. V2 — экспериментальный
путь, доступный только superuser через `/upload-v2/`. Расширение модели
не должно ломать V1 и не должно блокировать развитие V2. Стратегия:

### Принцип nullable + двухуровневое заполнение

Каждое новое поле модели добавляется как `null=True, blank=True`.
Парсеры заполняют ровно столько, сколько умеют:

| Категория полей                                | V1                                               | V2                                |
|------------------------------------------------|--------------------------------------------------|-----------------------------------|
| Существующие (`client_name`, `inn`, …)         | заполняет (как сейчас)                           | заполняет                         |
| Новые скалярные (адреса, ОГРН/КПП, даты лизинга, `insured_party`, частота рассрочки, основа возмещения) | оставляет `null` | заполняет, если нашёл             |
| `InsuredObject` (новая таблица)                | создаёт **один** объект из `vehicle_info` + `manufacturing_year` + `asset_status` | создаёт **N** объектов из payload |
| Поля объекта (`brand`, `model`, `vin`, `acquisition_cost`, …) | оставляет `null`                | заполняет из разобранной строки   |
| Structured `franchise.options[]`               | `null`                                           | заполняет из выбранных колонок    |
| `anti_theft_systems`                           | `null`                                           | заполняет                         |

V1 продолжает работать **без логических изменений**. Единственная
адаптация: после `read_insurance_request()` создаём один `InsuredObject`
из текущих полей, чтобы любая заявка (старая и новая) имела единое
представление объекта. Это 5–10 строк кода, не меняющих поведение
парсера.

### Существующие заявки в БД

После добавления `InsuredObject` нужна **data-миграция** через
`RunPython`: для каждой существующей `InsuranceRequest` создать один
`InsuredObject` со значениями `description = vehicle_info`,
`manufacturing_year = manufacturing_year`, `condition` нормализованный
из `asset_status`. Остальные поля объекта остаются `null`. Один прогон
— и историческая база совместима с новой схемой.

### Двойное представление объектов на время сосуществования парсеров

Пока V1 жив, держим **оба представления одновременно**:

- `InsuranceRequest.vehicle_info` остаётся в модели как «свод одной
  строкой» — старый код, шаблоны и письма продолжают читать его.
- `InsuredObject` — нормализованная таблица. Новый PDF, отчёты и
  фильтры читают её.

Когда V1 будет удалён, текстовое `vehicle_info` тоже убирается.

### Стратегия перехода

1. **Сейчас** — V1 дефолт, V2 за `/upload-v2/` для superuser.
2. **После расширения модели** — V1 продолжает работать, V2 заполняет
   больше полей. Оператор видит больше деталей в превью V2, может
   дозаполнить руками.
3. **Когда V2-покрытие стабильно ≥ V1 по всем полям** — сделать V2
   дефолтным путём загрузки, V1 оставить как fallback. Аудит уже
   показывает, что V2 либо обгоняет, либо равен V1 по почти всем полям.
4. **После периода стабильности** (несколько месяцев работы, accept-аудит)
   — удалить V1, освободить `core/excel_utils.py` от роли парсера.

### Что НЕ нужно делать

- Не пытаться «доучить» V1 извлекать новые поля. V1 — фиксированные
  координаты ячеек, его расширение хрупкое и съест время, которое
  лучше потратить на V2.
- Не делать новые поля `NOT NULL` — это сразу сломает V1-флоу и
  потребует фиктивных дефолтов.
- Не вынуждать UI требовать заполнения новых полей. Пустое поле
  — валидное состояние; оператор дозаполняет руками, когда нужно.

## Открытые вопросы

- Юридическая сторона: можно ли брокеру «упаковывать» данные лизинга в
  собственный документ под своим брендом — проверить условия договоров.
- Подпись/печать: ставить ли факсимильную подпись брокера на PDF.
- Многообъектные заявки (24 файла в корпусе с несколькими объектами,
  до 15 шт.): один PDF на заявку с таблицей объектов или PDF на объект.
- Совместимость со страховщиками: кому-то может быть критично сохранить
  и оригинал лизинга. Опция в форме отправки решает.
