# Implementation roadmap: модель → парсер → splitting

Дата: 2026-05-18.
Состояние: набросок для обсуждения. Каждый этап разберём детально перед стартом.

Документ описывает конкретные шаги реализации после уже сделанного **Этапа 1**
([поля партии в `InsuranceRequest`](../../insurance_requests/migrations/0032_add_batch_fields.py)).
Связанные документы: [own_insurance_request_form.md](own_insurance_request_form.md),
[coverage_audit.md](coverage_audit.md).

## Общая картина зависимостей

```
Этап 2 (модель)
   ├─ 2.1 Поля объекта  ─────────────┐
   ├─ 2.2 Реквизиты страхователя     │
   ├─ 2.3 Параметры сделки/страх.    ├──> Этап 3 (парсер) ──> Этап 4 (splitting view)
   └─ 2.4 Структурированные блоки    │
                                     │
   (2.2–2.4 независимы между собой,  │
    можно параллельно)               │
```

- **Этап 4 (splitting) разблокируется минимально после 2.1.** Этого
  достаточно, чтобы сёстры партии различались.
- **Этап 3 (парсер) тянет за собой все 2.x по мере готовности полей.**
  Можно расширять парсер инкрементально: чем больше полей в модели —
  тем больше парсер начинает заполнять.
- Каждый под-этап — отдельная миграция, отдельный коммит, отдельные тесты.

## Этап 2: расширение модели `InsuranceRequest`

Общие принципы для всех под-этапов:

- Все новые поля **`null=True, blank=True`** (см.
  [принцип совместимости](own_insurance_request_form.md#совместимость-парсеров-v1-и-v2)).
- Никаких `NOT NULL` и никаких дефолтных строк типа `'не указано'`.
- Поля хранятся **в `InsuranceRequest`**, отдельные таблицы не создаются
  (см. [архитектурный принцип «1 объект = 1 заявка»](own_insurance_request_form.md#ключевой-архитектурный-принцип-один-объект--одна-заявка)).
- Каждый под-этап после миграции добавляет smoke-тесты: создание записи
  со всеми полями, проверка `choices`, проверка дефолта `null`.
- V1 не трогаем; новые поля заполняет только V2 (на Этапе 3) или
  оператор руками через UI.

### Под-этап 2.1 — Поля объекта

**Цель.** Каждая заявка-сестра партии может различаться по объекту:
марка, модель, VIN, год, состояние, стоимость и т.п. Без этого
splitting на Этапе 4 бессмысленен — N сёстер будут полными клонами.

**Поля.**

| Поле                          | Тип                                                                 | Источник в JSON Schema v1               |
|-------------------------------|---------------------------------------------------------------------|------------------------------------------|
| `brand`                       | `CharField(max_length=128)`                                         | `insured_object.brand`                   |
| `model`                       | `CharField(max_length=255)`                                         | `insured_object.model`                   |
| `vin`                         | `CharField(max_length=32)`                                          | `insured_object.vin`                     |
| `serial_number`               | `CharField(max_length=64)`                                          | `insured_object.serial_number`           |
| `manufacturing_year_int`      | `PositiveSmallIntegerField` (1900–2100)                             | `insured_object.manufacturing_year`      |
| `condition`                   | `CharField(max_length=10, choices=['new','used','unknown'])`        | `insured_object.condition`               |
| `equipment_type`              | `CharField(max_length=128)`                                         | `insured_object.equipment_type`          |
| `power_or_capacity`           | `CharField(max_length=64)`                                          | `insured_object.power_or_capacity`       |
| `quantity`                    | `DecimalField(max_digits=10, decimal_places=2)`                     | `insured_object.quantity`                |
| `acquisition_cost_value`      | `DecimalField(max_digits=14, decimal_places=2)`                     | `insured_object.acquisition_cost.value`  |
| `acquisition_cost_currency`   | `CharField(max_length=3)` (`RUB`/`USD`/`EUR`)                       | `insured_object.acquisition_cost.currency` |

**Что разблокирует.** Этап 4 (splitting). Парсинг строки объекта на
Этапе 3.

**Что отсюда НЕ убираем.** Существующие текстовые поля
(`manufacturing_year` как `CharField`, `asset_status`, `vehicle_info`)
остаются для совместимости. V2 будет писать и в старые поля, и в новые
(дублирование). После полного перехода на V2 — старые поля можно
deprecate'нуть отдельно.

**Открытые вопросы.**

1. `condition`: использовать ли `unknown` как явное значение или
   просто `null` означает «не определено»? Я предлагаю `null`, без
   явного enum-значения `unknown` — это упрощает выборки.
2. `manufacturing_year_int` vs существующий `manufacturing_year`
   (string): сейчас V1 кладёт строку вида `'2024'`; иногда там
   слипшийся текст (`'2018 2018 2018'` при нескольких объектах).
   Дублируем в int нормализованно — но V1 продолжает писать в строку.
   После полного перехода строковое поле можно убрать. Когда?
3. `acquisition_cost_currency` — фиксируем 3 значения (`RUB`/`USD`/`EUR`)
   или оставляем свободным `CharField(max_length=3)`? В корпусе встречал
   только `руб` и `USD`. Я предлагаю свободный 3-letter ISO, валидация
   на стороне парсера/формы.
4. Что с `vehicle_info`: оставляем как есть (заполняется и V1, и V2
   — слиплённой строкой)? После перехода на V2 он становится
   избыточным.

### Под-этап 2.2 — Реквизиты страхователя

**Цель.** Хранить реквизиты, которые есть в Excel, но никуда не
попадают: юр./почтовый адрес, вид деятельности, ОГРН/КПП, дата рождения
ИП, дата подачи заявки.

**Поля.**

| Поле                | Тип                                       | Источник                                  |
|---------------------|--------------------------------------------|--------------------------------------------|
| `ogrn`              | `CharField(max_length=15)`                | (нет в схеме v1, **добавить в схему**)     |
| `kpp`               | `CharField(max_length=9)`                 | (нет в схеме v1, **добавить в схему**)     |
| `legal_address`     | `TextField`                               | `customer.legal_address` (research: 133/151) |
| `postal_address`    | `TextField`                               | `customer.postal_address` (86/151)         |
| `business_activity` | `TextField`                               | `customer.business_activity` (115/151)     |
| `birth_date`        | `DateField` (для ИП)                      | `customer.birth_date`                      |
| `submission_date`   | `DateField`                               | `request.submission_date` (148/151)        |

**Зависимости.** Параллельно — обновить
`docs/insurance_request_format_package/insurance_request_schema_v1.json`:
добавить `customer.ogrn` и `customer.kpp` как nullable.

**Открытые вопросы.**

1. ОГРН: 13 знаков (юр.лица) или 15 (ОГРНИП)? Стоит хранить как
   строку без валидации длины (max 15) или с двумя CharField
   `ogrn` / `ogrnip` отдельно? Я предлагаю один `ogrn(max_length=15)`,
   валидация чисел в форме/парсере.
2. Адреса — `TextField` (без лимита) или `CharField(max_length=512)`?
   В реальных заявках встречаются длинные адреса с переносами строк
   — лучше `TextField`.
3. `submission_date` или `request_date`? Это дата, когда лизинг подал
   нам заявку. В схеме v1 — `submission_date`. Согласовать имя поля.

### Под-этап 2.3 — Параметры сделки и страхования

**Цель.** Структурировать всё, что сейчас живёт либо в `insurance_period`
enum (одно из двух), либо вообще нигде не хранится.

**Поля.**

| Поле                                | Тип                                                            | Источник                                |
|-------------------------------------|----------------------------------------------------------------|------------------------------------------|
| `contract_start_date`               | `DateField`                                                    | `lease.contract_start_date`              |
| `contract_end_date`                 | `DateField`                                                    | `lease.contract_end_date`                |
| `insured_party`                     | `CharField(choices=['lessor','lessee','both','unknown'])`      | `lease.insured_party`                    |
| `period_start_date`                 | `DateField`                                                    | `insurance.period.start_date`            |
| `period_end_date`                   | `DateField`                                                    | `insurance.period.end_date`              |
| `period_months`                     | `PositiveSmallIntegerField`                                    | `insurance.period.months`                |
| `insured_sum_type`                  | `CharField(choices=['aggregate','non_aggregate'])`             | `coverage_terms.insured_sum_type`        |
| `indemnity_basis`                   | `CharField(choices=['with_depreciation','without_depreciation'])` | `coverage_terms.indemnity_basis`      |
| `guard_conditions`                  | `TextField`                                                    | `coverage_terms.guard_conditions`        |
| `property_location_right_holder`    | `CharField(choices=['lessee_owner','third_party_owner','unknown'])` | `coverage_terms.property_location_right_holder` (для имущества) |
| `premium_frequency`                 | `CharField(choices=['quarterly','semiannual','annual','custom'])` | `premium_payment.frequency`           |

**Зависимости.** Никаких новых. `insurance_period` (enum-строка) остаётся
для совместимости.

**Открытые вопросы.**

1. **`insurance_period` (старый enum) vs `period_start_date`/`period_end_date`/`period_months`.** Сейчас старое поле принимает `'1 год'` / `'на весь срок лизинга'`. После добавления дат старое становится избыточным. Дублируем (V2 пишет всё), потом deprecate'аем?
2. `insured_party = 'unknown'` или `null`? Аналогично `condition`.
3. `property_location_right_holder` нужен только для страхования имущества. Делать ли отдельно «applicable_for_property_only» или просто оставлять null для casco — пусть UI сам решает, что показывать?
4. `premium_frequency='custom'` — нужно ли отдельное текстовое поле `premium_frequency_comment` для описания нестандартной периодичности? В схеме v1 есть `premium_payment.comment` — добавить как `premium_frequency_comment`?

### Под-этап 2.4 — Структурированные блоки

**Цель.** Унести из `additional_data` JSON-блобов и из неструктурированного
текста те данные, которые имеют чёткую схему.

**Что добавляется.**

| Блок                         | Подход                                                                                  |
|------------------------------|------------------------------------------------------------------------------------------|
| **Structured-франшиза**      | `franchise_options` = `JSONField(default=list, blank=True)`. Список объектов `{kind, percent, amount_value, amount_currency}`. Старое `franchise_details` в `additional_data` мигрируем сюда. |
| **Анти-теф системы**         | `anti_theft` = `JSONField(default=dict, blank=True)`. Структура: `{alarm: {factory, additional, name_or_model}, immobilizer: {...}, mechanical_devices: [...], satellite_system: {required, brand, model, configuration}}`. |
| **Перевозка**                | Три flat-поля: `transportation_origin` (CharField), `transportation_destination` (CharField), `transportation_estimated_days` (PositiveSmallIntegerField). |
| **СМР**                      | `construction_work_description` (TextField). |

**Зависимости.** Под-этап 2.4 — самый «вязкий» по UX, потому что
JSON-поля требуют сложной формы редактирования в превью V2.
Можно отложить до завершения 2.1–2.3 и Этапа 3.

**Открытые вопросы.**

1. **Франшиза**: использовать JSONField или сделать **отдельную таблицу `FranchiseOption`** со связью FK к `InsuranceRequest`? Аргументы за JSON — простота, гибкость. Против — нельзя фильтровать заявки по %-у франшизы из ORM. Я склоняюсь к **JSONField**, потому что фильтрация по конкретному % редка, а структура вариантов меняется (для разных страховщиков).
2. **Анти-теф**: один JSON-объект или 4 отдельных JSON-поля (`anti_theft_alarm`, `anti_theft_immobilizer`, …)? Один объект экономит миграции, но усложняет частичный update. Предлагаю **один JSON**.
3. **Что делать со старыми `franchise_details` в `additional_data`?** Оставлять как fallback или мигрировать в новое поле через `RunPython`? Старые заявки V1 живут с `additional_data['franchise_details']` — лучше при чтении проверять оба источника, без forced миграции.

## Этап 3: парсер V2 — заполнение новых полей

**Цель.** V2 должен извлечь из Excel максимум того, что добавлено в
модель на Этапе 2.

### 3.1 — Разбор строки объекта (зависит от 2.1)

Сейчас V2 хранит объект как сырую строку:

```python
{"description": "LADA Largus KS045L 2024 б/у 78.05 1490000 руб", "year": "2024", "source": "..."}
```

Нужно разобрать на:

- `brand` / `model` — из начала строки до года.
- `vin` — поиск 17-символьного буквенно-цифрового кода (`[A-HJ-NPR-Z0-9]{17}` без I/O/Q).
- `serial_number` — для спецтехники (когда нет VIN).
- `manufacturing_year` (int) — уже извлекается, остаётся.
- `condition` — нормализовать «новое/новый/новая» → `new`, «б/у/бу/б-у» → `used`.
- `acquisition_cost.value` + `currency` — из столбцов M/N. По аудиту:
  M41 = `'Стоимость на момент приобретения'`, M43 = `'1490000'`, N43 = `'руб'`. То есть для парсера: ячейка M в строке объекта = value, N в той же строке = currency (нормализованная: `руб`→`RUB`).
- `equipment_type`, `power_or_capacity`, `quantity` — из соседних столбцов (L «Мощность двигателя» / L «колесная» / и т.п.).

**Открытые вопросы.**

1. Парсер строки объекта — regex-набор или попытка через NER/LLM? Я предлагаю regex+правила, LLM как fallback **только** для нераспознанных кейсов.
2. Если `brand`/`model` не удалось разделить, что сохранять? Целиком в `brand` и `model` оставлять null? Или наоборот? Я предлагаю: первое слово → `brand`, остальное до года → `model`; если разбить не удалось — всё в `model`.
3. Currency: список нормализаций — `руб` / `руб.` / `RUB` / `рублей` → `RUB`; `USD` / `$` / `долл` → `USD`. Согласовать.

### 3.2 — Реквизиты страхователя (зависит от 2.2)

Из аудита видно, что они в Excel есть, но V2 их не извлекает.
Нужно найти лейблы в реальной разметке (как с `asset_status`) и
добавить extractor'ы:

- Юр.адрес — лейбл «Юридический адрес» или «Юр. адрес».
- Почтовый адрес — «Почтовый адрес» / «Адрес для корреспонденции».
- Вид деятельности — «Вид деятельности» / «ОКВЭД».
- ОГРН/КПП — лейблы прямые.
- Дата рождения ИП — «Дата рождения».
- Дата подачи — обычно в шапке файла, формат разный.

**Открытые вопросы.**

1. ОГРН и ИНН могут быть в одной ячейке через слеш («ИНН/ОГРН»):
   тогда парсить из неё двух чисел.
2. Координаты или label-based? Я предлагаю **label-based с fallback на
   жёсткие ячейки**, по аналогии с фиксами `asset_status`.

### 3.3 — Параметры сделки и страхования (зависит от 2.3)

- Даты договора лизинга — обычно в шапке или верхней секции.
- `insured_party` — текст «Страхователем выступает: Лизингополучатель» и т.п.
- Период страхования — start/end даты, если есть; иначе оставляем enum-`kind`.
- `insured_sum_type` / `indemnity_basis` / `premium_frequency` — обычно в разделе «Условия страхования» с чекбоксами или текстом.

### 3.4 — Структурированные блоки (зависит от 2.4)

- Франшиза: V2 уже находит выбранную колонку, но не парсит сами значения. Добавить чтение значений % и абсолютных сумм рядом.
- Анти-теф: V2 сейчас использует маркеры только чтобы **пропускать** строки. Изменить логику: эти строки парсить отдельным извлекателем.
- Перевозка/СМР: вытащить origin/destination/description из соответствующих секций.

## Этап 4: splitting во view'е `/upload-v2/`

**Цель.** Когда парсер V2 находит N объектов в файле, создаётся
N записей `InsuranceRequest` — по одной на объект.

### Логика

1. **Парсинг** — `ExcelRequestParserV2.parse()` возвращает payload с
   массивом объектов (как сейчас) + новые объектные поля (после Этапа 3).
2. **Превью** — оператор видит **список из N будущих заявок**, может:
   - отредактировать любую (например, исправить распознанную марку);
   - отбраковать любую (например, если в Excel объект-дубль);
   - подтвердить всю партию или отдельные заявки.
3. **Создание** — на confirm:
   - генерируется один `source_batch_id` (UUID);
   - создаётся `N` записей `InsuranceRequest` с общими реквизитами
     (клиент, ИНН, тип/период, франшиза, ДФА, филиал, менеджер,
     даты лизинга, …);
   - в каждой записи уникальны только **объектные поля** (Этап 2.1)
     и индексы партии (`source_batch_id`, `item_no`, `item_count`).
4. **Attachment** — оригинальный Excel прикрепляется **к первой
   заявке партии** (или ко всем? см. открытый вопрос ниже).

### UI-изменения

- Шаблон превью `upload_excel_v2_preview.html` сейчас рендерит одну
  форму. Нужно превратить в **список форм с общими и индивидуальными
  полями**.
- В списке заявок (`request_list`) добавить визуальную группировку
  по `source_batch_id`: «партия из 3 заявок» / отображение сёстер
  рядом.
- В детальной странице заявки (`request_detail`) показывать
  «В этой партии ещё N заявок» со ссылками.

### Открытые вопросы

1. **Attachment**: один файл (оригинальный Excel) прикрепляется
   к одной заявке или копируется во все N? Один — экономит место,
   но непонятно, к какой именно «вешать». Я предлагаю **прикрепить
   ко всем** (через `RequestAttachment` — это разные строки, не
   дублирование файла на ФС, если использовать одинаковый
   `file.name`).
2. **Превью при N>5**: показывать сразу все или с пагинацией?
   В корпусе максимум 15, обычно 2–6.
3. **Пакетные действия в превью**: «применить ИНН ко всем» /
   «применить тип страхования ко всем». Полезно при ручном
   редактировании, но усложняет UI. Можно отложить.
4. **Что если оператор отбраковал все заявки в превью?**
   Сейчас V2-флоу создаёт хотя бы одну запись. После splitting'а
   возможен кейс «0 заявок» — отдавать ошибку или просто отменять
   загрузку?
5. **Письмо страховщику**: подтверждаю принцип «1 заявка → 1 письмо»,
   но в UI отправка идёт батчем (выбрать партию → отправить N писем).
   Это уже Этап 5+, но дизайн UX надо держать в голове.
