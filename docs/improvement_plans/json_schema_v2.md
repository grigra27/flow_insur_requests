# JSON Schema v2 — планируемая ревизия

Дата: 2026-05-20.
Статус: набросок для обсуждения. Сама схема ещё не создана; этот документ фиксирует, что и зачем меняем.

## Контекст

JSON Schema v1 (`docs/insurance_request_format_package/insurance_request_schema_v1.json`) была спроектирована «от идеала»: набор полей, которые мы хотели бы иметь в собственной форме страховой заявки. Однако аудит [178 файлов корпуса](coverage_audit.md#этап-31--разбор-строки-объекта) показал, что часть полей **системно отсутствует** в лизинговых заявках — парсер просто не из чего их извлекать.

Кроме того, после перехода к [принципу «1 объект = 1 заявка»](own_insurance_request_form.md#ключевой-архитектурный-принцип-один-объект--одна-заявка) часть полей схемы стала избыточной по другой причине: их семантика теряется после splitting'а партии.

## Что меняем в v2

### Удаляем из схемы

| Поле в v1                       | Fill rate в корпусе          | Почему удаляем                                                                            | Альтернатива                                                              |
|---------------------------------|------------------------------|-------------------------------------------------------------------------------------------|---------------------------------------------------------------------------|
| `insured_object.vin`            | 8.8% per-object, 6.7% per-file | В таблице объектов лизинга VIN не предусмотрен. Изредка встречается в свободном тексте описания. | Если страховщику нужно — оператор вводит руками; источник — ПТС/реестр.   |
| `insured_object.serial_number`  | 0% per-object                | Заводской/серийный номер для спецтехники в основной таблице объектов не указывается.       | Оператор вводит руками; источник — паспорт самоходной машины.             |
| `insured_object.quantity`       | 0.6% per-object              | После splitting'а 1 объект = 1 заявка → количество всегда `1`. Поле теряет смысл.          | Размер партии виден через `item_count` записи `InsuranceRequest`.         |

### Меняем структуру

| Изменение                                              | Причина                                                                                  |
|--------------------------------------------------------|-------------------------------------------------------------------------------------------|
| `insured_objects: [...]` (array) → `insured_object: {...}` (object) | Принцип «1 заявка = 1 объект». См. [архитектурный принцип](own_insurance_request_form.md#ключевой-архитектурный-принцип-один-объект--одна-заявка). |
| `schema_version: "alliance.insurance_request.v2"`      | Bump версии.                                                                              |

### Добавляем в схему

| Поле                       | Источник                          | Покрытие в корпусе (research) |
|----------------------------|-----------------------------------|--------------------------------|
| `customer.ogrn`            | Шапка заявки                      | требуется проверить            |
| `customer.kpp`             | Шапка заявки                      | требуется проверить            |
| `request.batch.batch_id`   | Партия (для отслеживания сёстер)  | внутренний UUID                |
| `request.batch.item_no`    | Позиция в партии                  | 1-based                        |
| `request.batch.item_count` | Размер партии                     | 1-based                        |

Поля `batch.*` опциональны — для одиночных заявок их можно не передавать (или ставить `item_count: 1`).

### Возможные кандидаты на пересмотр (требуют отдельной проверки)

Эти поля **сейчас не трогаем**, но при подготовке v2 стоит проверить их fill rate в корпусе:

- `coverage_terms.guard_conditions` — встречается ли в реальных заявках?
- `coverage_terms.property_location_right_holder` — только для имущества; если редко используется, можно опционализировать.
- `underwriting.anti_theft_systems.mechanical_devices[]` — может ли вообще быть массивом, или это один текст?

## Решения по открытым вопросам

1. **Поля `vin`/`serial_number`/`quantity` в БД — удалены.**
   На Этапе 2.1 они были добавлены, но фактически в источнике отсутствуют. Решено: миграция `0034_remove_vin_serial_quantity` удаляет их из `InsuranceRequest`, helpers `extract_vin` / `extract_serial_number` / `extract_quantity` убраны из парсера V2, payload объекта больше их не содержит. Поля сразу удаляем из истории, чтобы прод никогда их не увидел.

2. **Тайминг перехода на v2 — между Этапами 4 и 5.**
   Сразу после реализации splitting'а (Этап 4) и **перед** генерацией собственного PDF/JSON (Этап 5). К этому моменту структура batch уже будет в коде, и v2 формализует её в схеме.

3. **Backward compatibility не нужна.**
   V2 — внутренний контракт, наружу пока ничего не отдаём. После создания v2 удаляем v1 без переходного периода. Старые примеры `example_insurance_request_*.json` пересоздадим на v2.

4. **Парсер V2 не наполняет удалённые поля.**
   Решение «бонусная информация в payload» отклонено — нет смысла извлекать поля, которых в источнике нет. Helpers удалены вместе с полями модели.
