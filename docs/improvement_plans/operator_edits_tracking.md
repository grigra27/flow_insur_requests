# Трекинг ручных правок оператора в V2-флоу

Дата создания: 2026-06-05.
Статус: **реализовано** (все 4 фазы, ветка `feature/operator-edits-tracking`).

## Что реализовано

- **Фаза 1** — снимок «до» (`original_data`) и предвычисленный diff (`tracking`)
  в `additional_data['parser_v2']`; модуль `insurance_requests/edit_tracking.py`;
  блок «Ручные правки оператора» в карточке заявки.
- **Фаза 2** — страница `/<pk>/comparison/` с двухколоночным сравнением всех
  полей и ссылкой на исходный Excel; строго согласована с tracking.
- **Фаза 3** — денормализованный `manual_edits_count`, бейдж в списке (ведёт на
  сравнение), колонка/фильтр/таблица правок в Django admin (миграция 0041).
- **Фаза 4** — модель `RequestFieldEdit`, денормализованный `parser_confidence`
  (миграция 0042), дашборд `summaries:analytics_parser_edits` (топ полей, по
  филиалам/операторам/типам, доля заявок с правками, помесячная динамика).

Покрытие: юнит-тесты diff/сравнения + интеграционные тесты save-флоу, списка,
админки и дашборда. Полный затронутый прогон — 188 тестов зелёные.

## Задача

При загрузке Excel-заявки через новый V2-загрузчик («Загрузить заявку»)
парсер распознаёт данные и открывает страницу предпросмотра, где оператор
может вручную поправить любое поле перед сохранением. Нужен инструмент,
который показывает **разницу между тем, что распознал парсер из Excel, и
тем, что оператор сохранил вручную** — то есть какие поля пришлось
исправлять (где парсер ошибся) и что именно было изменено.

Двойная цель (подтверждена пользователем):

1. **Работа с конкретной заявкой** — открыл заявку, видишь, что оператор
   правил руками относительно распознанного Excel.
2. **Аналитика по парсеру** — какие поля операторы правят чаще всего, по
   филиалам/операторам/форматам, где парсер систематически слаб, как
   меняется уверенность парсера во времени.

Видимость нужна во всех четырёх местах: карточка заявки, отдельная
страница сравнения, метка в списке заявок, Django admin.

## Ключевой вывод: половина работы уже сделана

При сохранении заявки V2 в `additional_data['parser_v2']` уже пишутся два
слоя данных (`insurance_requests/views.py:170-182`,
функция `_build_parser_v2_additional_data`):

- `preview_fields` — финальные значения, которые оператор отправил,
  нажав «Сохранить» (`views.py:177`). Это слой **«после»**.
- `parsed_payload` — часть распознанных парсером данных
  (`views.py:152,176`). Это слой **«до»**.

То есть данные для сравнения уже частично копятся на каждой заявке. Это
**в основном задача отображения**, а не сбора данных с нуля.

## Нюанс данных: что сейчас сохраняется как «оригинал», а что нет

Парсер (`ExcelRequestParserV2.parse()`,
`insurance_requests/parsers/excel_v2/parser.py`) возвращает `ParserV2Result`,
где `data` содержит:

- на **верхнем уровне** — скалярные поля: `client_name`, `inn`,
  `dfa_number`, `branch`, `manager_name`, `insurance_type`,
  `insurance_period`, `franchise_type`, и т.д.;
- внутри — вложенный `parser_v2_payload` (объекты лизинга
  `insured_objects[]`, `object_grouping`, `raw_branch`,
  `franchise_details`, `application_format`/`application_type`).

При сохранении в `additional_data` копируется **только вложенный**
`parser_v2_payload` (`views.py:152` → `views.py:176`), а
**верхнеуровневые исходные значения скаляров не сохраняются** — они живут
только в сессионном черновике (`parser_v2_drafts[draft_id]`), который
удаляется после сохранения (`_pop_parser_v2_draft`).

### Следствие

Прямо сейчас можно построить diff:

- ✅ по объектам лизинга (марка/модель/стоимость) и по франшизе —
  данные в `parsed_payload`;
- ❌ по основным полям клиента (имя, ИНН, ДФА, филиал, менеджер) —
  исходное распознанное значение **не сохранено**, остался только финал
  (`preview_fields`).

### Что нужно для полного diff

Одна крошечная правка в `_build_parser_v2_additional_data`: сохранять
полный снимок исходного `data` (распознанные скаляры), а не только
`parser_v2_payload`. Буквально одна-две строки. После этого все будущие
заявки получат полный снимок «до/после».

> **Ретроспектива ограничена.** Для уже существующих заявок diff по
> скалярам восстановить нельзя (данных нет). По объектам лизинга и
> франшизе — частично можно. Полноценный diff работает «вперёд», с
> момента внедрения правки фазы 1.

## Варианты захвата данных (слой данных)

- **D1 — снимок «до/после» в `additional_data` (минимальный).**
  Дописать сохранение полного оригинала рядом с `preview_fields`. Diff
  считается на лету при отображении (сравнение двух dict). Без новой
  модели и миграций. Сложность: **очень низкая** (1–2 строки + функция
  сравнения).

- **D2 — снимок + предвычисленный diff.**
  То же, плюс при сохранении сразу считаем список изменённых полей и
  кладём готовый `tracking.operator_edits`. Удобно для аналитики без
  перебора всех заявок на лету. Сложность: **низкая**.

- **D3 — отдельная модель `RequestFieldEdit` / денормализованный счётчик.**
  Таблица: заявка, имя поля, исходное значение, новое значение, кто,
  когда. Плюс счётчик `manual_edits_count` на `InsuranceRequest` для
  бейджей в списке. Даёт нормальные SQL-запросы, фильтры в админке,
  агрегаты. Сложность: **средняя** (модель + миграция + запись + UI).

- **D4 — django-easy-audit (уже стоит).**
  Не подходит как основной путь: easy-audit видит изменения **после**
  создания заявки (правки в карточке), а наш кейс — правки **до**
  создания (на превью, до попадания объекта в БД). «Парсер сказал X,
  оператор сохранил Y» происходит внутри одного CREATE, easy-audit это
  не разворачивает.

## Варианты отображения (UX)

- **U1 — блок «Ручные правки» в карточке заявки.** Сворачиваемая секция,
  таблица только изменённых полей. Повседневный формат.

  ```
  ▼ Ручные правки оператора (изменено 3 поля)
  ┌────────────────┬─────────────────────┬─────────────────────┐
  │ Поле           │ Распознано из Excel │ Внёс оператор       │
  ├────────────────┼─────────────────────┼─────────────────────┤
  │ ИНН            │ 770123456 (8 цифр)  │ 7701234567 (10)     │
  │ Филиал         │ —                   │ Московский          │
  │ Год выпуска    │ 2021                │ 2022                │
  └────────────────┴─────────────────────┴─────────────────────┘
  Парсер: уверенность 78% · 2 предупреждения
  ```

- **U2 — отдельная страница/вкладка «Сравнение с Excel».** Все поля в две
  колонки (распознано / итог), изменённые подсвечены. Рядом ссылка на
  прикреплённый исходный Excel (`RequestAttachment`).

- **U3 — индикатор-бейдж в списке заявок.** Метка «✎ 3» (сколько полей
  правили) + цветовой индикатор уверенности парсера. Для надзора.

- **U4 — аналитический дашборд (в `summaries`).** Топ часто правимых
  полей, доля заявок с правками, по филиалам/форматам/операторам,
  динамика уверенности парсера. Требует D2 или D3. Под `Администраторы`.

- **U5 — Django admin.** Read-only inline/секция в админке заявки либо
  отдельный admin для `RequestFieldEdit` (если D3) с фильтрами. Минимум
  фронтенда.

## Выбранный объём и фундамент данных

Пользователь выбрал полный охват (обе цели + все четыре места). Раз нужны
бейджи в списке (U3), аналитика (U4) и фильтры в админке (U5) — все они
требуют **запрашиваемых** данных, не только JSON на одной заявке. Значит
фундамент — **D2 + D3**:

- **D2** обязателен в любом случае: только здесь хранится исходное
  распознанное значение каждого поля («до»). Плюс правка одной строкой,
  чтобы досохранять полный оригинал скаляров.
- **D3** нужен, чтобы список и аналитика не перебирали JSON всех заявок
  на лету. Минимально достаточно счётчика `manual_edits_count` на
  `InsuranceRequest` для бейджей + таблицы `RequestFieldEdit` для
  дашборда.

## Фазированный план

| Фаза | Состав | Результат | Оценка |
|---|---|---|---|
| **1** | D2: досохранять оригинал + функция-diff + блок U1 в карточке | Видно правки в конкретной заявке. Снимок начинает копиться | **0.5–1 день** |
| **2** | U2: отдельная страница сравнения в две колонки + ссылка на прикреплённый Excel | Подробный разбор заявки | +0.5 дня |
| **3** | D3: счётчик правок на заявке + бейджи U3 в списке + read-only секция/фильтр U5 в админке | Надзор: где парсер чаще ошибается, фильтрация | +1 день |
| **4** | Модель `RequestFieldEdit` + дашборд U4 в `summaries` | Аналитика: топ правимых полей, по филиалам/операторам, динамика уверенности | +1.5–2 дня |

Итого ~3.5–4.5 дня на весь объём. Фазы независимы и дают пользу по
отдельности — можно остановиться после любой.

## Открытые вопросы (уточнить на старте фазы 1)

- **Что считать «правкой».** Считать ли изменением случай «парсер вернул
  пусто → оператор заполнил» (по сути дозаполнение, а не исправление
  ошибки парсера)? Возможно, помечать такие записи отдельной категорией.
- **Нормализация перед сравнением.** Пробелы, регистр, формат дат/чисел —
  нормализовать, чтобы не ловить ложные диффы (напр. `«2021 »` vs
  `«2021»`).
- **Размещение дашборда.** Логично положить в `summaries` рядом с
  существующими аналитическими страницами, закрыть под `Администраторы`.
