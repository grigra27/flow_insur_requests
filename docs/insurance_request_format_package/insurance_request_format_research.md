# Исследование новой формы страховой заявки

Дата анализа: 12.05.2026.

## Что было проанализировано

Я просмотрел текущий загрузчик и корпус реальных заявок в `avtozayavka/real_requests`.

Обработан весь батч: 151 файл `.xls`, все прочитались. Внутри есть не только стандартные заявки, но и несколько краевых файлов вроде `акцепт...` и `свод котировок...`; это полезно как негативные примеры, которые новый формат должен отсеивать явной валидацией.

Текущий импорт расположен в `core/excel_utils.py`. Он читает данные по фиксированным ячейкам, а UI заставляет пользователя вручную выбрать тип заявки и формат: `legal_entity` / `individual_entrepreneur`, `casco_equipment` / `property`.

## Ключевые находки по корпусу

- Все 151 файла в реальном батче имеют старый формат `.xls`.
- 127 файлов похожи на заявки юрлиц, 24 файла - на заявки ИП.
- 148 файлов относятся к текущему формату КАСКО/спецтехника, 3 файла - к страхованию имущества.
- По виду страхования получилось: 114 КАСКО, 24 страхование спецтехники, 3 страхование имущества, 10 неразмеченных/краевых.
- 24 файла содержат больше одного объекта страхования. Максимум в одном файле - 15 объектов.
- 3 файла не выглядят как полноценные заявки с объектом страхования: два `акцепт...` и один `свод котировок...`.

## Почему текущий Excel хрупкий

Текущая форма для человека выглядит понятной, но для импорта это не таблица данных, а визуальный документ. Проблемы не случайные, они следуют из самой природы формы:

- Данные завязаны на координаты ячеек, например ИП сдвигает строки после 8-й строки.
- Объекты страхования сейчас сохраняются в одно текстовое поле `vehicle_info`, хотя фактически это массив объектов с годом, статусом, стоимостью, валютой и комментариями.
- Многообъектные заявки теряют структуру: в 24 файлах больше одного объекта.
- Условия франшизы представлены отметками в ячейках D/E/F; в новом формате их лучше передавать списком вариантов.
- Рассрочка/график оплаты переданы отметками рядом с текстовыми строками, что неоднозначно. Нужны явные `mode` и `frequency`.
- Признаки вроде автозапуска, КАСКО C/E, перевозки и СМР должны приходить явными boolean-полями, а не вычисляться по наличию текста в соседних строках.
- Страхование имущества отличается от КАСКО/спецтехники не только значениями, но и расположением строк объектов.

## Рекомендуемый формат

Основная рекомендация: **JSON-файл UTF-8 по JSON Schema**.

Почему JSON лучше всего подходит именно здесь:

- У заявки естественно вложенная структура: клиент, сделка, страхование, массив объектов, варианты франшизы, параметры андеррайтинга.
- JSON легко валидировать до создания заявки.
- Python/Django читает JSON без дополнительных зависимостей.
- Формат нормально хранит массив объектов, деньги, даты, boolean-поля и `null`.
- Можно версионировать контракт через `schema_version`.
- Заказчик может выгружать один файл на заявку или отправлять тот же payload в API.

Предлагаемый контракт:

- `schema_version` - версия формата, например `alliance.insurance_request.v1`.
- `request` - номер заявки, дата подачи, филиал, менеджер, тип заявителя, формат заявки.
- `customer` - клиент: тип, наименование, ИНН, адреса, дата рождения для ИП, вид деятельности.
- `lease` - даты договора лизинга, банк-кредитор, кто является страхователем.
- `insurance` - вид страхования, период, территория, условия покрытия, график оплаты, франшиза.
- `insured_objects` - массив объектов страхования.
- `underwriting` - дополнительные признаки: автозапуск, C/E, перевозка, СМР, ключи, ПТС/ПСМ, телематика, цели использования.
- `notes` - свободные комментарии, если они нужны.

Формальный вариант схемы лежит рядом:

- `insurance_request_schema_v1.json`
- `example_insurance_request_20213.json`
- `example_insurance_request_18022_multi_object.json`
- `example_insurance_request_20165_property_transportation.json`

## Проверка полноты относительно Excel-анкеты

После дополнительной сверки с реальными строками анкеты контракт v1 явно покрывает не только то, что сейчас забирает `ExcelReader`, но и поля, которые старый загрузчик частично теряет:

| Блок Excel-анкеты | Где отражен в JSON |
|---|---|
| Номер заявки, дата подачи, филиал, контактное лицо | `request` |
| Тип заявителя: юрлицо / ИП | `request.applicant_type`, `customer.type` |
| Клиент, ИНН, дата рождения ИП, адреса, вид деятельности | `customer` |
| Страхователь: лизингодатель / лизингополучатель | `lease.insured_party` |
| Сроки договора лизинга | `lease.contract_start_date`, `lease.contract_end_date` |
| Банк-кредитор | `lease.creditor_bank` |
| Вид страхования | `insurance.product` |
| Необходимый период страхования, включая нестандартные сроки | `insurance.period` |
| Территория страхования | `insurance.territory` |
| Страховая сумма, форма выплаты, условия охраны | `insurance.coverage_terms` |
| Правообладатель места расположения оборудования для имущества | `insurance.coverage_terms.property_location_right_holder` |
| Франшиза: без, процент, абсолютная сумма, оба варианта | `insurance.franchise.options` |
| Порядок уплаты премии: единовременно / рассрочка / периодичность | `insurance.premium_payment` |
| Все объекты страхования с годом, статусом, типом, стоимостью, валютой и комментарием | `insured_objects[]` |
| Цель использования | `underwriting.usage_purposes` |
| Автозапуск, ключи, ПТС/ПСМ, телематика | `underwriting` |
| Противоугонные системы | `underwriting.anti_theft_systems` |
| Дополнительные риски имущества: перевозка и СМР | `underwriting.additional_coverages` |

Оставшиеся спорные места не являются пропуском контракта, но требуют договоренности с заказчиком:

- В старых Excel часть полей содержит не атомарные данные, а общий текст. Например, марка/модель/VIN могут быть склеены в описании объекта. В JSON для них есть отдельные поля, но заказчик должен выгружать их отдельно, если они есть в его системе.
- В некоторых реальных файлах сами исходные данные неполные, например клиент указан просто как `ООО`. Новый формат это не исправит автоматически, но сможет валидировать и отклонять такие заявки до загрузки.
- Старый загрузчик местами принимает строки-ярлыки за данные. Например, многообъектная заявка может сдвинуть блок противоугонных систем ниже, а координатный парсер все равно смотрит старые строки. JSON устраняет этот класс ошибок.

## Альтернативы для заказчика

1. **JSON + JSON Schema - рекомендуемый вариант.**
   Лучший баланс строгости, простоты разработки и читаемости. Подходит и для файловой выгрузки, и для API.

2. **XML + XSD.**
   Имеет смысл, если система заказчика исторически умеет только XML-обмен или у них уже есть контур ЭДО/SOAP. По смыслу это та же иерархия, что JSON, но более многословная. Для нас тоже нормально, но разработка и поддержка будут тяжелее.

3. **ZIP-пакет с CSV/TSV.**
   Возможен, если заказчик может выгружать только плоские таблицы. Тогда нужно минимум три файла: `request.csv`, `objects.csv`, `options.csv`. Минусы: сложнее валидировать связи, неудобно передавать вложенные варианты франшизы и дополнительные параметры.

4. **Переходный строгий XLSX-шаблон без объединенных ячеек.**
   Это не целевой вариант, но хороший компромисс, если переговоры о JSON/XML не пройдут сразу. Шаблон должен быть не визуальной анкетой, а набором листов `request`, `customer`, `objects`, `coverage_options` с фиксированными заголовками колонок.

## Что просить у заказчика на переговорах

Минимальный запрос:

- Экспортировать одну заявку одним JSON-файлом UTF-8.
- Передавать все даты в ISO-формате `YYYY-MM-DD`.
- Передавать деньги как `{ "value": number, "currency": "RUB" }`.
- Передавать признаки только boolean-значениями: `true` / `false`.
- Передавать все предметы страхования массивом `insured_objects`, даже если объект один.
- Не использовать текстовые заглушки вроде `нет`, `-`, `н/д`; для отсутствующих значений использовать `null` или не передавать поле.
- Сохранять номер заявки заказчика в `request.request_number`.
- Для франшизы передавать массив вариантов, потому что в реальных файлах часто встречается "и с франшизой, и без".

Если заказчик не готов к JSON:

- второй вариант - XML + XSD;
- третий вариант - строгий XLSX с несколькими листами и без объединенных ячеек.

## Последствия для нашего проекта

Для внедрения можно не ломать старый загрузчик:

- добавить новый импорт `InsuranceRequestPayloadReader` для JSON;
- валидировать payload по `insurance_request_schema_v1.json`;
- маппить payload в текущую модель `InsuranceRequest`;
- на первом этапе класть полный исходный JSON в `additional_data`;
- позднее, если нужно, выделить `insured_objects` в отдельную модель, потому что текущий `vehicle_info` уже не хватает для многообъектных заявок.

Старый ExcelReader стоит оставить как fallback на переходный период.
